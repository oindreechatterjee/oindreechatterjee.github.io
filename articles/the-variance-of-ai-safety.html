The Variance of AI Safety, from Source Maps to Supply Chains

April 4, 2026 · 4 min read · (originally published on Medium)

Lessons from the Anthropic Leak and the Mercor Breach, both of which occurred within the past week.

I’ve spent a lot of time lately thinking about the relationship between AI safety and AI security. We often treat safety as the whole picture, but without security, safety guarantees fall apart. Looking at the Claude Code leak and the Mercor supply-chain breach, it’s becoming clear that the biggest threat to “safe” AI today isn’t just a rogue model, but rather a weak system.

The Claude Code Leak

The Anthropic leak is a good example of how a small technical mistake can turn into a real security issue. Claude Code source code was recently exposed through an npm packaging error involving a source map, revealing roughly 512,000 lines of unobfuscated TypeScript across about 1,900 files.

I learned that this incident isn’t simply a secret-loss issue, but it’s a roadmap. When attackers can see the orchestration logic, permissions, build details, and execution paths, they can study the system for weak points. They do not need to guess how the agent is pieced together, but can directly inspect it. This matters especially in AI systems, because if an attacker learns how the agent handles tools, permissions, and hidden behaviors, what is stopping them from looking for ways to bypass guardrails or exploit assumptions in the workflow? The code leak becomes a guide for attack surface discovery.

More surprising to me: the leak itself became bait. Zscaler reported that a fake “Claude Code leak” repository was used to distribute malware, including Vidar and GhostSocks. That means the incident did not stop at exposing internal code. It also created a new trust signal that attackers could weaponize.

The Mercor Breach

In this case, the issue was tied to LiteLLM, a popular gateway used to manage access to multiple AI models. Attackers compromised a separate tool in the pipeline to get publishing access, then pushed two poisoned versions of LiteLLM to PyPI. The malicious packages were live for less than an hour, but LiteLLM is downloaded millions of times a day, and thousands of companies were hit, one of them being Mercor.

What I learned from this breach is that many AI companies are dependent on middleware they do not fully control. That is convenient for shipping fast, but it also creates a dangerous concentration point. Mercor is a useful example because it shows how a startup can be affected even if the core model itself isn’t the problem. Because Mercor’s clients include Meta and OpenAI, the breach is a reminder that supply chain risk flows upward—a compromised middleware layer doesn’t just hurt the startup, it touches everyone connected to it. The vulnerability was not in the foundation model, but it was in the system used to connect the model to the product.

This made me realize that open-source infrastructure introduces a shared-trust problem. Once a tool like LiteLLM becomes a common dependency, it also becomes an attractive target. If that middle layer is poisoned, every company built on top of it inherits the risk.

What these leaks say about AI safety

The Anthropic leak and the Mercor breach illustrate that AI safety is layered. A useful way to think about it is as an onion. At the center is the model itself. Around that is the harness—the code that controls how the model behaves, what tools it can use, and how it is deployed. The outer layer is the broader supply chain: external packages, gateways, vendor integrations, and other third-party infrastructure.

The Claude leak hit the inner layer by exposing internal implementation details. The Mercor breach hit an outer layer by compromising a shared dependency. Different attack paths, yet same lesson. The entire system has to be secure for the AI product to be safe.

These recent incidents have revealed that the AI safety conversation needs to broaden beyond alignment, robustness, and misuse. Because in the short term, a lot of the real-world risk comes from much more basic security failures. Exposed source code, compromised dependencies, leaked credentials, and weak operational controls create a more pressing security issue in the present.

The idea that AI safety is only about model behavior is too narrow. In practice, safety starts with the surrounding infrastructure. If the harness is exposed, or the supply chain is compromised, the model’s behavior is no longer the only thing that matters.

These two incidents have honestly changed how I view the scope of AI safety. I used to think of it mostly as a math and ethics problem (making sure a model is aligned and won’t go rogue!). But looking at these cases, I’m realizing that safety is also a more practical, daily struggle with security.
